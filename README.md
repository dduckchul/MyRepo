# 비개발자를 위한 Git 사용법

이 프로젝트는 개발자가 아닌 사람도 Git과 GitHub를 부담 없이 이해하고 사용할 수 있도록 돕기 위한 안내서입니다.

파일 이름에 `최종`, `진짜최종`, `진짜진짜최종`을 붙이던 방식에서 벗어나, 변경 이력을 남기고 여러 사람과 안전하게 협업하는 방법을 차근차근 정리합니다.

## 목차

## 1. 이 문서는 누구를 위한 문서인가요?

- Git을 처음 접하는 기획자, 디자이너, PM, 작가, 연구자
- 개발 지식 없이 문서나 파일의 변경 이력을 관리하고 싶은 사람
- GitHub에서 파일을 수정하거나 협업해야 하는 사람
- 개발자와 함께 일하면서 Git 용어를 이해하고 싶은 사람

## 2. Git이 왜 필요한가요?

- 파일 이름에 `최종`, `진짜최종`, `수정본`을 계속 붙이지 않아도 되는 이유
- 누가, 언제, 무엇을 바꿨는지 확인하는 방법
- 이전 상태로 되돌아갈 수 있다는 것의 의미
- 여러 사람이 같은 프로젝트를 안전하게 수정하는 방식

## 3. Git과 GitHub는 뭐가 다른가요?

- Git: 내 컴퓨터에서 파일의 변경 이력을 관리하는 도구
- GitHub: Git으로 관리한 파일을 온라인에서 공유하고 협업하는 서비스
- 둘을 함께 사용할 때 생기는 장점

## 4. 꼭 알아야 할 기본 개념

- 저장소 Repository
- 커밋 Commit
- 브랜치 Branch
- 푸시 Push
- 풀 Pull
- 머지 Merge
- 충돌 Conflict

## 5. 처음 시작하기

- GitHub 계정 만들기
- GitHub Desktop 설치하기
- 저장소 만들기
- 파일을 열고 수정하는 법
- 변경 내용을 저장하는 기본 흐름

## 6. GitHub Desktop으로 작업하기

- GitHub Desktop 설치하기
- 저장소 초기화 하기
- 파일 수정하기
- 변경 내용 확인하기
- 커밋하기

## 7. 협업할 때의 기본 흐름

- 최신 내용 가져오기
- 새 브랜치 만들기
- 내 작업 진행하기
- 커밋으로 변경 내용 저장하기
- 작업 내용을 합치기

## 8. 자주 생기는 상황과 해결법

- 최신 파일이 아닌 상태에서 작업했을 때
- 충돌 Conflict이 발생했을 때
- 잘못 수정한 내용을 되돌리고 싶을 때
- Pull Request에서 수정 요청을 받았을 때

## 9. 절대 하면 안 되는 것

- 모르는 파일을 함부로 삭제하지 않기
- 남의 브랜치에 직접 작업하지 않기
- 큰 파일을 아무 생각 없이 올리지 않기
- 커밋 메시지를 비워두지 않기
- 이해하지 못한 경고창을 무시하고 넘기지 않기

## 10. 실습 예제

- README 파일 한 줄 수정하기
- 자기소개 파일 추가하기
- 브랜치를 만들어서 수정하기

## 11. 용어 사전

Git과 GitHub에서 자주 나오는 용어를 쉬운 말로 정리합니다.

- Repository
- Commit
- Branch
- Push
- Pull
- Merge
- Pull Request
- Conflict
